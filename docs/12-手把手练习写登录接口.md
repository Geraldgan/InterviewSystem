# 手把手练习写登录接口

这份文档的目标不是“直接帮你把代码写完”，而是带你按这个项目现有风格，自己练手写一个最基础的登录接口。

我会尽量用你现在已经熟悉的这套结构来讲：

- `Controller`
- `Service`
- `Repository`
- `Entity`
- `DTO`

也就是说，这份文档更像一份“练习路线图 + 落地步骤说明”，而不是现成答案。

---

## 1. 先明确这次练手的目标

我们假设你想做的是一个最常见的登录接口：

```text
POST /api/auth/login
```

前端传：

```json
{
  "username": "gerald",
  "password": "123456"
}
```

后端返回：

```json
{
  "userId": 1,
  "username": "gerald",
  "displayName": "Gerald Gan",
  "token": "xxxxxx"
}
```

这次练手你要学会的，不只是“能返回成功”，而是把整条链路自己搭出来：

1. 前端请求怎么定义
2. 数据库表怎么设计
3. Repository 怎么查用户
4. Service 怎么校验密码
5. Controller 怎么接请求并返回响应
6. 异常怎么处理
7. 测试怎么写

---

## 2. 先说一句很重要的话

### 这次练手建议先不要一上来就把难度拉满

如果你现在是第一次自己写登录接口，我建议你先分两层来练：

### 第一层：先练“接口结构”

你先把这些打通：

- 请求能进来
- 能查到用户
- 能校验用户名密码
- 能返回成功 / 失败结果

这一步先不纠结最完整的权限体系。

### 第二层：再练“安全细节”

等第一层通了，再继续加：

- 密码加密
- JWT
- 拦截器 / 过滤器
- 角色权限
- 刷新 token

也就是说，先别把“登录接口”想得太大。

你现在这次练手，最适合做的是：

```text
一个最小可工作的登录接口
```

如果你要按当前项目风格来写，建议先规划这些文件。

### Entity

例如：

### Controller

例如：

- `backend/src/main/java/com/interview/backend/controller/AuthController.java`

如果你照这个路径去写，代码结构会和你现在已经看过的：

- `PositionProfileService`
- `QuestionSetService`

保持同一套风格。

## 4. 第一步：先别写代码，先把接口 contract 写清楚

你写任何接口前，第一步都不是先敲 `Controller`，而是先想清楚：

### 这个接口的输入是什么

登录接口最小输入一般是：

- 用户名
- 密码

所以 `LoginRequest` 你可以先设计成：

```text
username
password
```

### 这个接口的输出是什么

最小输出一般是：

- 用户 ID
- 用户名
- 展示名
- token

也可以先不返回太多，只返回：

- `token`
- `userId`

### 失败时返回什么

你也要先想清楚失败场景：

- 用户不存在
- 密码错误
- 账号被禁用
- 参数为空

这一层想清楚之后，你再动手写代码，会顺很多。

---

## 5. 第二步：先设计数据库表，而不是先写 Service

登录接口的核心是“校验用户身份”，所以你至少得先有一张用户表。

建议你先自己回答这几个问题：

### 用户表需要哪些字段

最小版本通常包括：

- `id`
- `username`
- `password_hash`
- `display_name`
- `active`
- `created_at`
- `updated_at`

### 用户名要不要唯一

一般答案是：

- 要唯一

因为登录时通常通过用户名查用户。

### 密码能不能明文存

答案是：

- 不能

即使你只是练手，也建议你从一开始就建立正确习惯：

- 数据库存的是加密后的密码
- 登录时拿用户输入密码去做比对

### 表名叫什么

建议简单明确，比如：

- `user_accounts`

---

## 6. 第三步：写 `Entity` 时你要想什么

假设你准备写：

- `UserAccount.java`

你在写实体时，不要只想着“把字段抄到 Java 类里”，而是要想：

### 这个实体是给谁用的

实体是给：

- JPA / Hibernate
- 数据库存取

用的。

它不是直接给前端返回的。

### 你要给哪些字段加约束

例如：

- `username` 要唯一
- `passwordHash` 不能为空
- `displayName` 不能为空或可空，这取决于你设计
- `active` 需要默认值

### 是否继承 `BaseEntity`

当前项目已经有：

- `BaseEntity`

所以你完全可以复用它的：

- `id`
- `createdAt`
- `updatedAt`

这样你的 `UserAccount` 只需要关心用户自己的字段。

---

## 7. 第四步：写 Repository 时先只做最小能力

对于登录接口，你的 Repository 一开始不需要设计太多方法。

最关键的通常就一个：

```text
按用户名查用户
```

也就是说，你可以先只想这类方法：

```java
findByUsername(...)
```

或者如果你也有启用状态：

```java
findByUsernameAndActiveTrue(...)
```

你现在应该重点练的是：

- “我知道登录要按什么条件查用户”

而不是一开始就写一堆 Repository 方法。

---

## 8. 第五步：DTO 怎么拆，为什么不要直接拿 Entity 接请求

登录接口一般至少要有两个 DTO。

### 请求 DTO

- `LoginRequest`

它负责接前端参数。

一般会有：

- `username`
- `password`

### 响应 DTO

- `LoginResponse`

它负责返回前端结果。

一般会有：

- `userId`
- `username`
- `displayName`
- `token`

为什么不用 `UserAccount` 直接接收和返回？

因为：

- `Entity` 是数据库模型
- `DTO` 是接口模型

数据库模型和接口模型通常不应该混在一起。

比如：

- 你绝不能把 `passwordHash` 原样返回给前端

这就是 DTO 很重要的原因。

---

## 9. 第六步：真正的核心在 Service

登录接口最值得你练的，就是 `Service`。

因为登录的业务逻辑都在这里。

你可以先自己给 `AuthService` 列一个伪流程：

```text
1. 校验请求参数
2. 根据 username 查用户
3. 如果用户不存在，报错
4. 如果用户被禁用，报错
5. 校验密码是否正确
6. 如果密码错误，报错
7. 生成 token
8. 返回 LoginResponse
```

这就是登录接口最核心的业务骨架。

如果这一步你能自己写顺，说明你已经开始掌握“业务层思维”了。

---

## 10. 登录接口里最容易犯错的地方

这里我提前告诉你几个最常见的坑。

### 1. 直接查到用户就登录成功

不对。

登录不是“查到用户就行”，而是：

- 查到用户
- 再校验密码

### 2. 把数据库密码当明文比较

不建议。

正确思路应该是：

- 注册时存加密后的密码
- 登录时用密码比对工具比较

### 3. 把“用户不存在”和“密码错误”写得过于详细

如果是正式系统，通常不会把错误提示写得太具体。

例如正式项目常常只返回：

```text
用户名或密码错误
```

这样更安全。

但你现在练手阶段，为了方便调试，可以先分开写，之后再收敛。

### 4. 在 Controller 里写太多逻辑

不建议。

Controller 最好只做：

- 接请求
- 调 Service
- 返回结果

不要把“查用户、比密码、生成 token”全堆在 Controller 里。

---

## 11. 第七步：token 先怎么练最合适

这里你要注意，你现在练的是“写接口”，不是一次性完成完整认证系统。

所以 token 这块建议这样练。

### 第一阶段练手做法

你可以先“假装有 token 体系”，先返回一个字符串。

比如：

- UUID
- 一个简单的随机字符串

目的不是上生产，而是先打通：

```text
登录成功后返回凭证
```

### 第二阶段再升级

等你第一版写顺以后，再继续升级：

- JWT
- token 过期时间
- 刷新 token
- 用户信息放 claims

如果你一开始就上 JWT + Spring Security，全局难度会一下抬很高。

---

## 12. 第八步：Controller 应该写到什么程度

你的 `AuthController` 最好只做下面几件事：

1. 接 `POST /api/auth/login`
2. 接收 `LoginRequest`
3. 调用 `authService.login(request)`
4. 返回 `LoginResponse`

如果你发现 Controller 里开始出现这些逻辑：

- 查数据库
- 比对密码
- 生成 token
- 拼业务错误信息

那通常说明你写重了，应该把逻辑往 Service 移。

---

## 13. 第九步：异常怎么设计

登录接口一定会遇到失败场景。

所以你最好在真正写之前就想清楚：

### 哪些错误属于业务错误

例如：

- 用户不存在
- 密码错误
- 用户被禁用

### 哪些错误属于参数错误

例如：

- 用户名为空
- 密码为空

### 哪些错误属于系统错误

例如：

- 数据库连接失败
- 程序异常

你现在项目里已经有全局异常处理风格了，所以练手时也最好沿用：

- 业务错误抛自定义异常
- 统一交给全局异常处理器返回

这样风格会和项目整体一致。

---

## 14. 第十步：测试要怎么练

如果你真想通过这个练手把能力练扎实，建议至少写两类测试。

### 1. Service 层思维测试

你要验证：

- 查不到用户时会不会报错
- 密码不对时会不会报错
- 登录成功时能不能返回正确结果

### 2. Controller 层接口测试

你要验证：

- `POST /api/auth/login` 能否正常返回 `200`
- 参数错误时会不会返回 `400`
- 用户名密码错误时返回的错误结构对不对

也就是说，登录接口不是“代码能跑就结束”，而是：

```text
业务逻辑要能测
接口表现也要能测
```

## 15. 推荐你按这个顺序自己动手

下面这个顺序最适合练手。

### 第 1 步

先写接口说明，不写代码：

- 路径是什么
- 请求体长什么样
- 返回体长什么样
- 失败场景有哪些

### 第 2 步

先设计用户表字段。

### 第 3 步

写 `UserAccount` 实体。

### 第 4 步

写 `UserAccountRepository`，只先做“按用户名查用户”。

### 第 5 步

写 `LoginRequest` 和 `LoginResponse`。

### 第 6 步

写 `AuthService`，先只实现这条主链：

```text
查用户 -> 校验密码 -> 返回结果
```

### 第 7 步

写 `AuthController`，只负责接请求和调用 Service。

### 第 8 步

补异常处理。

### 第 9 步

补测试。

### 第 10 步

最后再考虑：

- token 规范化
- 安全增强
- JWT / Spring Security

## 16. 你练手时可以照着问自己的问题

为了避免“看懂了但不会写”，你可以边写边问自己下面这些问题。

### 关于请求

- 前端传哪些字段才够登录？
- 哪些字段必须必填？

### 关于数据库

- 我准备按什么字段查用户？
- 这个字段需要唯一吗？
- 密码在数据库里应该怎么存？

### 关于 Service

- 如果查不到用户，我准备怎么处理？
- 如果密码不对，我准备怎么处理？
- 登录成功后，我要返回哪些字段？

### 关于 Controller

- 这层是不是只在做“接参数 + 调 Service + 返回结果”？
- 我有没有把业务逻辑错误地写进 Controller？

### 关于安全

- 我是不是避免了明文密码存库？
- 我返回给前端的内容里有没有泄露 `passwordHash`？

## 17. 一个非常适合你的练手目标

我建议你第一次练这个接口时，不要追求“完整认证系统”，而是先给自己定一个很清晰的小目标：

```text
自己独立写出一个最小可工作的登录接口，
能完成：
参数接收
数据库查用户
密码校验
返回 token
失败返回
```

如果这个目标你能独立完成，其实已经很不错了。

因为这说明你已经开始真正掌握后端接口开发的基本链路。

---

## 18. 练手完成后你应该复盘什么

你写完后，建议你自己复盘这几件事：

1. 哪一层写起来最费劲
2. 你最容易把逻辑写错到哪一层
3. 你是不是已经能自然分清：
   - Entity
   - DTO
   - Repository
   - Service
   - Controller
4. 你对“接口 = 业务逻辑 + 数据访问 + 返回结构”这件事是不是更有感觉了

如果你复盘完这些，再去写注册接口、修改密码接口，会顺很多。

---

## 19. 一句话版练习路线

如果你只记最短版本，就记这句：

```text
先设计请求和表结构，
再写 Entity / Repository / DTO，
然后把登录核心逻辑写进 Service，
最后用 Controller 暴露出去。
```
