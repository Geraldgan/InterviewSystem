# 登录接口练手模板

这份文档是给你“真正开始动手”时用的模板。

它的目标不是直接替你实现登录接口，而是帮你把工作拆成一个你可以自己照着完成的结构。

你可以把它当成：

- 文件清单
- 每个文件该写什么
- 哪层代码最简单
- 哪层代码最核心
- 写的时候要注意哪些坑

这份模板默认你练的是一个“最小可工作的登录接口”：

```text
POST /api/auth/login
```

---

## 1. 这次练手建议的目标范围

建议你第一版只覆盖这些内容：

- 用户名 + 密码登录
- 从数据库查用户
- 校验密码
- 返回一个登录成功响应
- 登录失败能返回业务错误

建议你第一版先不要强行加满这些内容：

- Spring Security 全套
- JWT 完整拦截链
- 刷新 token
- RBAC 权限
- 多端登录态管理

你这次的练习目标应该是：

```text
自己独立打通 Controller -> Service -> Repository -> MySQL 这条登录链路
```

## 3. 你可以按这个顺序动手

建议顺序：

1. 先写接口设计稿
2. 再写用户表实体
3. 再写 Repository
4. 再写请求和响应 DTO
5. 再写 Service
6. 再写 Controller
7. 再补异常
8. 最后补测试

不要反过来。

最容易把自己绕晕的做法就是：

- 一上来先写 Controller
- 写着写着才发现 DTO 没想清楚
- 再回去改 Entity
- 最后整个链路全乱了

## 4. 文件模板：`UserAccount.java`

文件位置：

- `backend/src/main/java/com/interview/backend/entity/UserAccount.java`

这一层的职责：

- 映射数据库表
- 只负责数据库字段
- 不负责接口返回

你可以先按这个思路写：

```java
package com.interview.backend.entity;

import jakarta.persistence.Column;
import jakarta.persistence.Entity;
import jakarta.persistence.Table;

@Entity
@Table(name = "user_accounts")
public class UserAccount extends BaseEntity {

    @Column(nullable = false, unique = true, length = 80)
    private String username;

    @Column(nullable = false, length = 255)
    private String passwordHash;

    @Column(nullable = false, length = 120)
    private String displayName;

    @Column(nullable = false)
    private boolean active = true;

    // getter / setter
}
```

这一层你要重点想清楚：

- `username` 是否唯一
- 密码字段存明文还是 hash
- 是否需要 `active`

最关键提醒：

- 数据库里不要存明文密码

## 5. 文件模板：`UserAccountRepository.java`

文件位置：

- `backend/src/main/java/com/interview/backend/repository/UserAccountRepository.java`

这一层的职责：

- 用最少的方法完成查用户

第一版建议只写最关键的方法：

```java
package com.interview.backend.repository;

import java.util.Optional;

import com.interview.backend.entity.UserAccount;
import org.springframework.data.jpa.repository.JpaRepository;

public interface UserAccountRepository extends JpaRepository<UserAccount, Long> {

    Optional<UserAccount> findByUsernameAndActiveTrue(String username);
}
```

这个方法的意义是：

- 按用户名查
- 只查启用中的账号

你练手时要先体会的是：

- Repository 不是乱写一堆方法
- 而是围绕当前业务需要，先写“刚好够用”的查询

## 6. 文件模板：`LoginRequest.java`

文件位置：

- `backend/src/main/java/com/interview/backend/dto/LoginRequest.java`

这一层的职责：

- 接前端请求参数

你可以先这样设计：

```java
package com.interview.backend.dto;

import jakarta.validation.constraints.NotBlank;

public record LoginRequest(
    @NotBlank(message = "用户名不能为空")
    String username,

    @NotBlank(message = "密码不能为空")
    String password
) {
}
```

这里你要重点学的是：

- 请求参数校验应该尽量前置
- 不要等进了 Service 再发现用户名是空的

## 7. 文件模板：`LoginResponse.java`

文件位置：

- `backend/src/main/java/com/interview/backend/dto/LoginResponse.java`

这一层的职责：

- 返回给前端登录成功后的结果

第一版可以简单一点：

```java
package com.interview.backend.dto;

public record LoginResponse(
    Long userId,
    String username,
    String displayName,
    String token
) {
}
```

你这里要刻意记住：

- 响应 DTO 里不要出现 `passwordHash`
- 不要直接把 Entity 原样返回给前端

## 8. 文件模板：`LoginFailedException.java`

文件位置：

- `backend/src/main/java/com/interview/backend/exception/LoginFailedException.java`

这一层的职责：

- 表达“登录失败”这种业务错误

模板可以很简单：

```java
package com.interview.backend.exception;

public class LoginFailedException extends RuntimeException {

    public LoginFailedException(String message) {
        super(message);
    }
}
```

你第一版不用设计太复杂。

核心是：

- 把“业务失败”从普通系统异常里分开

## 9. 文件模板：`AuthService.java`

文件位置：

- `backend/src/main/java/com/interview/backend/service/AuthService.java`

这一层最重要。

它的职责是：

- 查用户
- 校验密码
- 生成 token
- 返回登录结果

先不要追求完整实现，你先按这个骨架写：

```java
package com.interview.backend.service;

import com.interview.backend.dto.LoginRequest;
import com.interview.backend.dto.LoginResponse;
import com.interview.backend.entity.UserAccount;
import com.interview.backend.exception.LoginFailedException;
import com.interview.backend.repository.UserAccountRepository;
import org.springframework.stereotype.Service;
import org.springframework.util.StringUtils;

@Service
public class AuthService {

    private final UserAccountRepository userAccountRepository;

    public AuthService(UserAccountRepository userAccountRepository) {
        this.userAccountRepository = userAccountRepository;
    }

    public LoginResponse login(LoginRequest request) {
        String username = normalize(request.username());
        String password = request.password();

        UserAccount user = userAccountRepository.findByUsernameAndActiveTrue(username)
            .orElseThrow(() -> new LoginFailedException("用户名或密码错误"));

        if (!passwordMatches(password, user.getPasswordHash())) {
            throw new LoginFailedException("用户名或密码错误");
        }

        String token = generateToken(user);

        return new LoginResponse(
            user.getId(),
            user.getUsername(),
            user.getDisplayName(),
            token
        );
    }

    private String normalize(String value) {
        return StringUtils.hasText(value) ? value.trim() : null;
    }

    private boolean passwordMatches(String rawPassword, String passwordHash) {
        // TODO: 这里先接入密码校验逻辑
        return false;
    }

    private String generateToken(UserAccount user) {
        // TODO: 第一版可以先返回简单 token，后续再升级 JWT
        return "";
    }
}
```

你这层最需要自己完成的，主要是两块：

- `passwordMatches(...)`
- `generateToken(...)`

也就是说，真正值得你练手的，不是类壳子，而是这两个具体点。

## 10. `AuthService` 写的时候你要自己补的核心逻辑

当你写 `AuthService` 时，你要强迫自己回答下面这几个问题。

### 1. 用户名查不到怎么办

推荐：

- 抛 `LoginFailedException`

### 2. 密码不对怎么办

推荐：

- 同样抛 `LoginFailedException`

### 3. 为什么推荐统一提示“用户名或密码错误”

因为正式系统里不建议区分得太细。

比如不要轻易返回：

- “用户不存在”
- “密码错误”

这样对外暴露信息太多。

练手阶段你也可以先拆开写，但建议最后收敛成统一提示。

### 4. token 第一版怎么做

建议：

- 第一版先返回一个简单字符串
- 比如 UUID
- 先把“接口链路”练通

等你第一版跑通后，再升级：

- JWT
- 过期时间
- claims

## 11. 文件模板：`AuthController.java`

文件位置：

- `backend/src/main/java/com/interview/backend/controller/AuthController.java`

这一层的职责：

- 接收请求
- 调 Service
- 返回结果

模板如下：

```java
package com.interview.backend.controller;

import com.interview.backend.dto.LoginRequest;
import com.interview.backend.dto.LoginResponse;
import com.interview.backend.service.AuthService;
import jakarta.validation.Valid;
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
@RequestMapping("/api/auth")
public class AuthController {

    private final AuthService authService;

    public AuthController(AuthService authService) {
        this.authService = authService;
    }

    @PostMapping("/login")
    public LoginResponse login(@Valid @RequestBody LoginRequest request) {
        return authService.login(request);
    }
}
```

你在 Controller 里要刻意保持克制：

- 不要查数据库
- 不要比密码
- 不要拼业务逻辑

Controller 最好只做“转发”。

## 12. 如果你想加一个初始化测试用户，可以怎么练

第一版为了方便联调，你可以选择加一个“启动时插入测试用户”的组件。

思路和 `SeedDataInitializer` 很像：

- 如果用户表为空
- 就插入一个测试账号

例如：

```text
username = test
passwordHash = 经过加密后的密码
displayName = 测试用户
active = true
```

但这里一定要注意：

- 不要把明文密码直接写进数据库

即使是测试用户，也建议先用正确习惯。

## 13. 第一版密码校验怎么练最合适

这里我给你一个练习建议，而不是直接替你实现。

### 最推荐的练习顺序

#### 第 1 步

先学会“密码不能明文存库”这个原则。

#### 第 2 步

选一个简单可靠的密码编码器思路，例如：

- `BCrypt`

#### 第 3 步

在注册或初始化测试用户时，把密码先编码后再入库。

#### 第 4 步

登录时不要比较：

```java
rawPassword.equals(passwordHash)
```

而要走密码编码器的匹配逻辑。

你这次练手的核心目标是：

- 建立正确的安全习惯

而不是“先用最省事但错误的方式跑通”。

---

## 14. 第一版 token 怎么练最合适

这次练手建议这样分层。

### 第一版

先做：

- 登录成功返回一个 token 字符串

你可以先把它理解成：

- 一个占位凭证

### 第二版

再升级成：

- JWT

### 第三版

再考虑：

- token 过期
- 刷新 token
- 权限校验

你现在最重要的是：

- 把登录业务链路写顺

## 18. 一个你可以参考的伪代码总览

如果你想要一眼看全局，可以用这段伪代码记住：

```text
POST /api/auth/login
    -> AuthController.login(request)
    -> AuthService.login(request)
        -> repository.findByUsernameAndActiveTrue(username)
        -> 校验用户是否存在
        -> 校验密码是否匹配
        -> 生成 token
        -> 返回 LoginResponse
```

这条链你只要能自己独立敲出来，练习就已经非常值了。

## 19. 这次练手你最该关注什么

如果你只盯着“最后能不能返回 token”，很容易忽略真正重要的东西。

你更应该关注的是：

- 你有没有把职责分层分清楚
- 你是不是知道每一层为什么存在
- 你是不是能自己从接口需求倒推到 Entity / Repository / Service / Controller

这才是这次练手真正的收获。
