# 登录接口密码校验与测试模板

这份文档专门回答你现在最关键的两个问题：

- `passwordMatches(...)` 到底应该怎么实现
- `generateToken(...)` 第一版到底应该怎么实现

以及：

- 写完以后你应该怎么测接口
- 先用什么方式测最省力
- 测试时要覆盖哪些场景

这份文档仍然按“练手第一版”的思路来写。

也就是说：

- 先把登录链路练通
- 再慢慢升级安全细节

---

## 1. 先说结论：推荐你怎么实现

如果你现在是第一次自己把登录接口写完整，我建议分成两个阶段。

### 第一阶段：先练通登录链路

- `passwordMatches(...)`：用 `BCryptPasswordEncoder`
- `generateToken(...)`：先返回一个简单 token，比如 `UUID`

### 第二阶段：再升级成更真实的方案

- `passwordMatches(...)`：继续保留 `BCrypt`
- `generateToken(...)`：升级成 `JWT`

也就是说，你现在最推荐的组合是：

```text
密码校验用 BCrypt
token 先用 UUID 占位
```

这是最适合“练手第一版”的平衡点。

---

## 2. 为什么 `passwordMatches(...)` 不应该自己手写加密逻辑

很多人第一次写登录接口时，会想这样做：

```java
return rawPassword.equals(passwordHash);
```

这是不对的。

因为：

- 数据库里不应该存明文密码
- 登录时也不应该自己随便拼一个加密算法

更推荐的做法是：

- 用成熟的密码编码器
- 例如 `BCryptPasswordEncoder`

原因很简单：

- 这是 Spring 体系里最常见、最成熟的做法之一
- 它专门就是干“密码 hash 和密码匹配”这件事的

所以你不要把 `passwordMatches(...)` 理解成：

```text
自己写加密算法
```

而应该理解成：

```text
调用成熟的密码编码器做匹配
```

---

## 3. 第一版 `passwordMatches(...)` 模板

### 推荐依赖

如果你要走这个方案，通常需要有密码编码器支持。

最常见做法是加 Spring Security 的 crypto 能力。

你可以理解成：

- 不上整套 Spring Security
- 先只用它的密码加密工具

第一版你在 `AuthService` 里可以按这个思路写。

### 模板思路

```java
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import org.springframework.security.crypto.password.PasswordEncoder;

@Service
public class AuthService {

    private final UserAccountRepository userAccountRepository;
    private final PasswordEncoder passwordEncoder;

    public AuthService(
        UserAccountRepository userAccountRepository,
        PasswordEncoder passwordEncoder
    ) {
        this.userAccountRepository = userAccountRepository;
        this.passwordEncoder = passwordEncoder;
    }

    private boolean passwordMatches(String rawPassword, String passwordHash) {
        return passwordEncoder.matches(rawPassword, passwordHash);
    }
}
```

这里最关键的是：

- `rawPassword` 是用户输入的明文密码
- `passwordHash` 是数据库里存的加密结果
- `matches(...)` 会负责做正确的比较

你不用自己去“解密密码”。

因为密码校验的正确思路不是“解密”，而是：

- 拿用户输入密码
- 走同样的编码匹配逻辑
- 判断是否一致

## 4. `PasswordEncoder` Bean 模板

如果你用上面的写法，通常还要给 Spring 提供一个 `PasswordEncoder` Bean。

比如你可以单独放一个配置类：

- `backend/src/main/java/com/interview/backend/config/SecuritySupportConfig.java`

模板：

```java
package com.interview.backend.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import org.springframework.security.crypto.password.PasswordEncoder;

@Configuration
public class SecuritySupportConfig {

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

这样之后：

- `AuthService` 可以直接注入 `PasswordEncoder`

## 5. 测试用户密码应该怎么存

这里非常重要。

如果你想插一个测试用户，你不能这样存：

```text
password_hash = 123456
```

你应该存的是：

- `123456` 编码之后的结果

也就是说：

- 登录输入的是 `123456`
- 数据库里存的是一串 BCrypt hash

你可以这样理解：

```text
注册 / 初始化用户时：encode
登录时：matches
```

这个心智模型很重要。

---

## 6. 如果你要插测试用户，模板怎么想

假设你做一个启动时插入测试用户的初始化类，你的思路应该是：

1. 注入 `UserAccountRepository`
2. 注入 `PasswordEncoder`
3. 如果用户表为空，就插一个测试账号
4. 存进去的密码必须先 `encode(...)`

伪代码模板：

```java
UserAccount user = new UserAccount();
user.setUsername("test");
user.setDisplayName("测试用户");
user.setPasswordHash(passwordEncoder.encode("123456"));
user.setActive(true);
repository.save(user);
```

注意：

- `encode(...)` 只在“存密码”阶段做
- 登录时不要再 `encode(rawPassword).equals(dbHash)`
- 登录时应该用 `matches(...)`

## 7. 第一版 `generateToken(...)` 模板

如果你现在只是练手第一版，我建议你不要一开始就强上 JWT。

### 最简单可行模板

```java
import java.util.UUID;

private String generateToken(UserAccount user) {
    return UUID.randomUUID().toString();
}
```

这版的意义不是“生产可用”，而是：

- 登录成功后先返回一个凭证字符串
- 把接口链路练通

这一步你要练的是：

- Service 成功后能生成一个值
- Response 能把这个值带回前端

而不是立刻把认证系统做满。

---

## 8. 为什么第一版 token 可以先简单一点

因为你现在练的是：

- 后端接口开发能力
- 分层能力
- 查库能力
- 登录主流程

不是一步到位练：

- 完整认证框架
- 权限体系
- 登录态管理

所以第一版 token 先做成 UUID，是合理的。

等你这版顺了，再升级：

- JWT
- 过期时间
- claims
- 刷新 token

---

## 9. 如果你想升级成 JWT，思路是什么

先说结论：

- 第一版不必上
- 第二版非常值得学

JWT 版 `generateToken(...)` 的核心思路是：

1. 准备一个密钥
2. 把用户 ID、用户名等信息写进 claims
3. 设置过期时间
4. 签名生成 token

你现在不用急着实现，只要先知道：

- UUID 版是“练习用”
- JWT 版是“进阶版”

## 10. 第一版 `AuthService` 推荐长什么样

你现在的 `AuthService` 可以先朝这个结构靠近：

```java
public LoginResponse login(LoginRequest request) {
    String username = normalize(request.username());
    String password = request.password();

    UserAccount user = userAccountRepository.findByUsernameAndActiveTrue(username)
        .orElseThrow(() -> new LoginFailedException("用户名或密码错误"));

    if (!passwordMatches(password, user.getPasswordHash())) {
        throw new LoginFailedException("用户名或密码错误");
    }

    String token = generateToken(user);

    return new LoginResponse(
        user.getId(),
        user.getUsername(),
        user.getDisplayName(),
        token
    );
}
```

你现在真正要补的核心就两个点：

- `passwordMatches(...)`
- `generateToken(...)`

别的骨架你基本已经有了。

## 11. 写完后先怎么测试最省力

我建议你按下面顺序测试。

### 第一层：先用 `curl` 或 Postman 手测

这是最省力、最直观的。

你应该先验证：

- 接口能不能进 Controller
- 数据能不能进 Service
- 登录成功能不能拿到响应
- 失败时能不能返回预期错误

### 第二层：再补自动化测试

等手测通了，再写：

- `AuthServiceTests`
- `AuthControllerTests`

这样你不会一边调业务一边和测试细节打架。

---

## 12. 手动测试前你要准备什么

在你测登录接口前，先确认这几件事。

### 1. 后端已经跑起来

例如：

```bash
curl -s http://127.0.0.1:8081/api/health
```

能返回正常状态。

### 2. 数据库里已经有测试用户

例如你已经插入了一个：

- `username = test`
- 原始密码 = `123456`

但数据库存的是经过 `encode(...)` 后的 `passwordHash`

### 3. 你已经有登录接口

例如：

```text
POST /api/auth/login
```

---

## 13. `curl` 测试模板

假设你的接口是：

```text
POST /api/auth/login
```

那你可以先这样测成功场景：

```bash
curl -s -X POST "http://127.0.0.1:8081/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{
    "username": "test",
    "password": "123456"
  }'
```

你预期看到的是类似：

```json
{
  "userId": 1,
  "username": "test",
  "displayName": "测试用户",
  "token": "xxxxxx"
}
```

---

## 14. 失败场景 `curl` 测试模板

### 用户名或密码错误

```bash
curl -s -X POST "http://127.0.0.1:8081/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{
    "username": "test",
    "password": "wrong-password"
  }'
```

你要观察：

- HTTP 状态码是什么
- 返回的错误 JSON 结构是什么
- 是否符合你项目现在的全局异常风格

### 参数为空

```bash
curl -s -X POST "http://127.0.0.1:8081/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{
    "username": "",
    "password": ""
  }'
```

你要观察：

- 是否返回 `400`
- 校验信息是不是你预期的

## 15. 你手测时至少要覆盖这 5 个场景

建议你按这个清单测。

### 场景 1

- 用户存在
- 密码正确
- 登录成功

### 场景 2

- 用户存在
- 密码错误
- 登录失败

### 场景 3

- 用户不存在
- 登录失败

### 场景 4

- 用户被禁用
- 登录失败

### 场景 5

- 请求参数为空
- 参数校验失败

这 5 个场景一旦都通了，你的第一版登录接口就已经很像样了。

---

## 16. `AuthControllerTests` 你应该怎么写

Controller 测试主要测接口表现。

你重点可以写这些断言：

- 成功登录时返回 `200`
- 返回 JSON 里包含 `token`
- 参数为空时返回 `400`
- 登录失败时返回对应错误状态码

你可以把它理解成：

```text
Controller 测的是“接口对外表现”
```

而不是深入测密码算法本身。

---

## 17. `AuthServiceTests` 你应该怎么写

Service 测试重点测业务逻辑。

建议覆盖：

- 能根据用户名查到用户
- 密码匹配时返回成功响应
- 密码不匹配时抛异常
- 查不到用户时抛异常
- 被禁用用户不能登录

你可以把它理解成：

```text
Service 测的是“脑子”
Controller 测的是“脸”
```

也就是：

- Service 测业务逻辑
- Controller 测 HTTP 表现

---

## 18. 写完后你怎么判断“第一版已经合格了”

你可以用下面这个标准给自己做判断。

### 结构上合格

- 有 Entity
- 有 Repository
- 有 DTO
- 有 Service
- 有 Controller
- 有异常

### 逻辑上合格

- 能查用户
- 能校验密码
- 能返回 token
- 能区分成功和失败

### 测试上合格

- 至少手测成功和失败场景
- 至少有 1 个 Controller 测试
- 至少有 1 个 Service 测试

如果这些都做到了，你这次练手就已经非常有价值了。

结合你当前阶段，我最推荐你这样落地：
