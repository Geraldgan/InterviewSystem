# 登录接口自检清单

这份文档是给你在“第一版登录接口已经写出来之后”用的。

它的作用不是教你从 0 开始写，而是帮你做最后一轮自查：

- 哪些地方已经做对了
- 哪些地方还漏了
- 哪些点现在可以先接受
- 哪些点最好不要省略

你可以把它当成提交代码前的一张 checklist。

---

## 1. 先确认：你的第一版目标是不是清楚

先问自己一句：

```text
我现在是在做“最小可工作的登录接口”，
还是已经准备做“完整认证系统”？
```

如果你现在还是第一版练手，那这份清单默认你要达到的是：

- 用户名密码登录能跑通
- 能查数据库
- 能校验密码
- 能返回 token
- 成功失败路径都能走通

如果你还没到这一步，就先别着急追 JWT、拦截器、权限系统。

---

## 2. 接口定义自检

先检查接口 contract 有没有清楚。

### 路径

- [ ] 你是否已经明确接口路径，例如 `POST /api/auth/login`

### 请求体

- [ ] 你是否已经明确请求体字段
- [ ] 是否至少包含 `username` 和 `password`
- [ ] 请求字段命名是否统一、清晰

### 响应体

- [ ] 你是否已经明确成功响应结构
- [ ] 是否返回了前端真正需要的字段
- [ ] 是否避免把数据库内部字段直接暴露给前端

### 失败响应

- [ ] 你是否想清楚失败时返回什么
- [ ] 参数错误和业务错误是否能区分

如果你在这一步还答不清，说明接口设计还没完全稳住。

---

## 3. Entity 自检

检查你的用户实体，例如 `UserAccount`。

### 表映射

- [ ] 是否加了 `@Entity`
- [ ] 是否指定了合理的表名，比如 `user_accounts`
- [ ] 是否正确继承了 `BaseEntity`，或者你是否自己定义了主键和时间字段

### 字段设计

- [ ] 是否有 `username`
- [ ] 是否有 `passwordHash`
- [ ] 是否有 `displayName`
- [ ] 是否有 `active`

### 字段约束

- [ ] `username` 是否设置为唯一
- [ ] `passwordHash` 是否不能为空
- [ ] `active` 是否有默认值

### 安全点

- [ ] 你是否避免了把明文密码存到数据库字段里

这里最重要的一条是：

```text
数据库里应该存 passwordHash，不应该存 password 明文。
```

---

## 4. Repository 自检

检查你的 `UserAccountRepository`。

### 基本结构

- [ ] 是否继承了 `JpaRepository<UserAccount, Long>`

### 查询方法

- [ ] 是否已经有“按用户名查用户”的方法
- [ ] 如果你有启用状态，是否用了类似 `findByUsernameAndActiveTrue(...)`

### 复杂度控制

- [ ] 你是否只保留了“当前登录接口真正需要的方法”
- [ ] 你是否避免一开始就写很多暂时用不到的 Repository 方法

这一层的目标是：

- 刚好够用

而不是：

- 一次写满所有未来可能用到的方法

---

## 5. Request DTO 自检

检查你的 `LoginRequest`。

### 字段

- [ ] 是否包含 `username`
- [ ] 是否包含 `password`

### 参数校验

- [ ] 是否给必填字段加了校验注解
- [ ] 用户名为空时是否会被拦住
- [ ] 密码为空时是否会被拦住

### 职责边界

- [ ] DTO 是否只负责接收数据，而没有混入业务逻辑

---

## 6. Response DTO 自检

检查你的 `LoginResponse`。

### 返回字段

- [ ] 是否包含 `userId`
- [ ] 是否包含 `username`
- [ ] 是否包含 `displayName`
- [ ] 是否包含 `token`

### 安全性

- [ ] 是否没有返回 `passwordHash`
- [ ] 是否没有直接把整个 Entity 原样返回给前端

这一层最关键的一条是：

```text
返回给前端的结构，应该是你有意识设计过的，而不是“数据库里有什么就返回什么”。
```

---

## 7. Service 自检

这一层是最核心的。

检查你的 `AuthService.login(...)` 是否完整。

### 主流程

- [ ] 是否先拿到了请求里的 `username` 和 `password`
- [ ] 是否做了必要的标准化处理，比如 `trim`
- [ ] 是否先根据用户名查用户
- [ ] 查不到用户时是否会抛业务异常
- [ ] 是否检查了用户启用状态
- [ ] 是否做了密码匹配
- [ ] 密码不对时是否会抛业务异常
- [ ] 登录成功后是否生成了 token
- [ ] 是否返回了 `LoginResponse`

### 分层

- [ ] Service 里是否承担了核心业务逻辑
- [ ] 你是否没有把这些逻辑丢进 Controller

### 错误处理

- [ ] 登录失败时是否走的是业务异常，而不是随便 `return null`

如果你这里还有：

- `return null`
- `if else` 混乱
- 把数据库异常和业务异常揉在一起

那说明这一层还值得继续整理。

---

## 8. `passwordMatches(...)` 自检

这一块单独拿出来检查。

### 正确思路

- [ ] 你是否用了成熟密码编码器，而不是自己写加密算法
- [ ] 你是否使用了类似 `PasswordEncoder.matches(rawPassword, passwordHash)` 的思路

### 错误思路排查

- [ ] 你是否避免了 `rawPassword.equals(passwordHash)`
- [ ] 你是否避免了登录时自己重新 encode 后直接字符串比较

### 数据一致性

- [ ] 你数据库里的测试用户密码，是否真的是 encode 后存进去的

这里最容易出问题的一条是：

```text
代码写得看起来对，但数据库里的 passwordHash 其实不是同一套编码器生成的。
```

如果登录总失败，优先排查这个。

---

## 9. `generateToken(...)` 自检

如果你现在还是第一版练手，token 这里先不要过度内耗。

### 第一版目标

- [ ] 登录成功时是否能稳定返回一个 token 字符串

### 如果你现在用的是 UUID

- [ ] 是否已经足够支撑“接口练手第一版”

### 如果你已经上 JWT

- [ ] 是否有签名
- [ ] 是否设置了过期时间
- [ ] 是否放入了必要 claims

### 不要做的事

- [ ] 是否避免把明文密码塞进 token
- [ ] 是否避免把过多敏感信息塞进 token

如果你现在还在第一版，最重要的是：

- token 能稳定返回

不是：

- token 体系已经做到生产级

---

## 10. Controller 自检

检查你的 `AuthController`。

### 基本结构

- [ ] 是否暴露了 `POST /api/auth/login`
- [ ] 是否用了 `@RequestBody`
- [ ] 是否用了 `@Valid`

### 逻辑边界

- [ ] Controller 里是否只做了“接请求 + 调 Service + 返回响应”
- [ ] 是否没有在 Controller 里写查数据库逻辑
- [ ] 是否没有在 Controller 里写密码校验逻辑
- [ ] 是否没有在 Controller 里写 token 生成逻辑

如果你发现 Controller 已经写成了“半个 Service”，说明这层职责跑偏了。

---

## 11. 异常处理自检

登录接口一定要检查失败路径。

### 业务异常

- [ ] 用户不存在时是否抛业务异常
- [ ] 密码错误时是否抛业务异常
- [ ] 用户被禁用时是否抛业务异常

### 参数异常

- [ ] 参数为空时是否由校验机制拦住

### 返回风格

- [ ] 错误响应格式是否和当前项目已有异常风格一致

### 提示语

- [ ] 是否避免把错误提示写得过于泄露信息

例如正式系统里，更推荐：

- “用户名或密码错误”

而不是：

- “用户名不存在”
- “密码错误”

练手阶段你可以先分开，但建议你心里知道正式做法是什么。

---

## 12. 测试数据自检

在你说“接口不通”之前，先确认测试数据本身对不对。

### 测试用户

- [ ] 数据库里是否真的有这个用户名
- [ ] 用户是否处于 `active=true`
- [ ] 密码 hash 是否是用正确方式生成的

### 测试参数

- [ ] 你发请求时用户名拼对了吗
- [ ] 你发请求时密码是原始密码，而不是 hash

这一层非常常见的坑是：

- 代码没错
- 但你喂进去的测试数据是错的

---

## 13. 手动联调自检

写完第一版后，建议先手测。

### 基础联调

- [ ] 后端是否已经启动
- [ ] 登录接口路径是否正确
- [ ] `Content-Type: application/json` 是否正确

### 成功场景

- [ ] 正确用户名 + 正确密码时是否返回成功
- [ ] 是否拿到了 token

### 失败场景

- [ ] 错误密码时是否返回失败
- [ ] 不存在用户时是否返回失败
- [ ] 参数为空时是否返回 `400`

如果手测都没通，不要急着写自动化测试，先把链路跑通。

---

## 14. 自动化测试自检

手测通了以后，再检查测试覆盖。

### Service 测试

- [ ] 是否覆盖成功登录
- [ ] 是否覆盖查不到用户
- [ ] 是否覆盖密码错误
- [ ] 是否覆盖禁用用户

### Controller 测试

- [ ] 是否覆盖 `POST /api/auth/login`
- [ ] 是否覆盖参数校验失败
- [ ] 是否覆盖登录失败接口表现

### 回归价值

- [ ] 你的测试是否真的能在以后改代码时帮你发现回归问题

---

## 15. 安全习惯自检

这是练手阶段也值得先建立的习惯。

- [ ] 数据库里不存明文密码
- [ ] 不把 `passwordHash` 返回给前端
- [ ] 不把密码写进日志
- [ ] 不把密码写进 token
- [ ] 错误提示不过度泄露内部信息

如果这几条你都做到了，你已经在用比较健康的方式练习了。

---

## 16. 分层设计自检

你写完以后，最后一定要看这一层。

### Entity

- [ ] 是否只负责数据库映射

### Repository

- [ ] 是否只负责数据访问

### DTO

- [ ] 是否只负责收发接口数据

### Service

- [ ] 是否承担了核心业务逻辑

### Controller

- [ ] 是否只负责暴露 HTTP 接口

如果你每层都守住了职责，你这次练习就很成功。

---

## 17. 写完后你最应该复盘的问题

这几个问题非常值得你自己回答一下。

- [ ] 我最不熟的是哪一层？
- [ ] 我最容易把逻辑写错到哪一层？
- [ ] 我现在是否比之前更清楚 Controller / Service / Repository 的分工？
- [ ] 我对“接口不是只写一个 Controller 方法”这件事是不是更有感觉了？
- [ ] 如果现在让我再写一个“注册接口”，我是否已经知道该怎么拆层？

如果这些问题你能答得越来越顺，说明你真的在进步。

---

## 18. 提交代码前最后一轮检查

如果你准备把这次练手代码提交到 Git，最后再看一遍这几条：

- [ ] 能正常启动项目
- [ ] 登录接口能成功调用
- [ ] 至少一个成功场景手测通过
- [ ] 至少一个失败场景手测通过
- [ ] 没有把明文密码写进代码或数据库
- [ ] 没有把敏感字段返回给前端
- [ ] 代码分层没有明显混乱

这 7 条如果都能打勾，你的第一版就已经很不错了。

---

## 19. 一句话版验收标准

如果你只想记最短版本，就记这句：

```text
能查到用户，
能正确比密码，
能返回 token，
成功失败都能走通，
并且没有把密码明文乱放。
```

---

## 20. 你下一步最适合做什么

如果你已经按这份清单检查完了，下一步最值得做的是二选一：

### 路线 A

继续补强当前登录接口：

- 上 `BCrypt`
- 上 JWT
- 补更多测试

### 路线 B

趁热打铁再写一个接口：

- 注册接口
- 获取当前用户信息接口
- 修改密码接口

如果你想把“后端接口开发”真正练熟，我更推荐路线 B。

因为：

- 再写一个新接口
- 比只在一个接口里反复打磨
- 更能帮助你建立完整的分层感觉
