# 后端入门阅读顺序

这份文档是给你做“后端阅读导航”的。

如果你现在的状态是：

- 后端能跑起来
- 但对 `Spring Boot / Bean / Service / Repository` 还不够熟
- 想按一个不绕路的顺序把项目后端看明白

那就按下面这 3 份文档来读。

---

## 推荐阅读顺序

### 第 1 份：先看项目是怎么启动起来的

- [10-SpringBoot启动流程与Bean依赖图.md](/Users/geraldgan/Documents/GeraldGan/实践/InterviewSystem/docs/10-SpringBoot启动流程与Bean依赖图.md)

先看这一份的原因是：

- 你要先知道 `Spring Boot` 在项目里到底负责什么
- 要先知道 `Bean` 是什么、为什么对象会被自动创建
- 要先知道 `Controller / Service / Repository` 为什么能自动串起来
- 要先知道 `SeedDataInitializer` 为什么会自动执行

这份文档主要解决的问题是：

- 项目从 `main()` 到 ready 的启动流程
- Spring 容器是什么
- Bean 是怎么被创建和注入的
- 启动时哪些东西会自动发生

如果这一步没看懂，后面看 `Service` 很容易感觉“这些对象怎么凭空就能用”。

---

### 第 2 份：再看一个最简单的 Service

- [08-PositionProfileService详解.md](/Users/geraldgan/Documents/GeraldGan/实践/InterviewSystem/docs/08-PositionProfileService详解.md)

这份是最适合入门的业务层例子，因为它足够简单。

它主要讲：

- `GET /api/positions` 是怎么走到数据库的
- `Controller -> Service -> Repository -> Entity -> MySQL` 这条链路怎么理解
- Spring Data JPA 为什么不写 SQL 也能查表
- `PositionProfileResponse` 为什么要和实体分开

这份文档最适合你建立“查数据库接口”的基本心智模型。

你可以把它理解成：

- 先学会最简单的一条查询链路

---

### 第 3 份：最后看最核心的复杂 Service

- [09-QuestionSetService详解.md](/Users/geraldgan/Documents/GeraldGan/实践/InterviewSystem/docs/09-QuestionSetService详解.md)

这一份是进阶版，也是项目最核心的一份业务文档。

它主要讲：

- 生成题集请求是怎么被处理的
- 为什么要先补默认值、查岗位画像、再组 `GenerationCommand`
- `QuestionGenerationService` 是怎么接进来的
- 生成结果为什么要先转实体再落库
- 为什么一次 `save(questionSet)` 就能把题集和题目一起存进去
- 列表接口和详情接口又是怎么查出来的

你可以把这份理解成：

- 在“简单查询链路”基础上，再看“复杂业务链路”

---

## 为什么是这个顺序

这个顺序不是随便排的，而是按理解成本从低到高来排：

1. 先理解 Spring Boot 怎么把项目启动起来
2. 再理解一个最简单的查询型 Service
3. 最后再理解一个生成 + 查库 + 落库 + DTO 转换都包含的复杂 Service

也就是说，这个顺序本质上是：

```text
先理解框架
-> 再理解简单业务
-> 最后理解复杂业务
```

## 读完后你应该能回答什么问题

如果你把这 3 份按顺序读完，理论上你应该能比较稳地回答这些问题：

- Spring Boot 为什么不用我手动 `new` Controller 和 Service
- Bean 是怎么被发现和注入的
- `SeedDataInitializer` 为什么会自动执行
- Repository 为什么不写 SQL 也能查数据库
- 一个请求为什么会先经过 Controller，再进 Service
- Service 为什么不直接把实体原样返回给前端
- 为什么一次保存 `QuestionSet` 能把 `InterviewQuestion` 一起存进去

如果这些问题你都能说清楚，说明你对这个项目的后端骨架已经入门了。
